# Windows 桌面 Todo 插件 PRD

## Problem Statement

用户需要一个常驻 Windows 桌面的 Todo 工具。它不应该像传统 Todo 软件一样每次都打开完整窗口，而应该像 macOS「提醒事项」桌面插件一样，平时安静地显示当前任务，在需要时通过快捷键或底部隐藏图标进入编辑状态。

用户明确希望整体视觉参考 macOS Reminders：浅色界面、圆角窗口、柔和阴影、左侧固定列表导航、右侧任务区域、Today / Scheduled / All 三个入口，并使用本地提供的 SF Pro 字体文件，让 Windows 上的桌面插件具备接近 macOS 的字体观感和界面气质。

## Solution

构建一个 Windows 桌面 Todo 小组件应用。应用默认以无边框、圆角、轻量化桌面插件窗口呈现，并固定在桌面层的最底部，像桌面组件一样位于普通应用窗口后方。用户按下全局快捷键，或点击底部隐藏图标 / 系统托盘图标后，进入编辑模式。编辑模式下可以添加、修改、完成、删除、搜索任务，也可以管理日期和基础设置。退出编辑模式后，应用恢复为低打扰的桌面插件状态。

视觉上参考 macOS Reminders，但不直接复制 Apple 专有图标或系统资源。应用从用户提供的 SF Pro 字体中提取英文、数字和拉丁标点，并为中文内置独立的简体中文无衬线字体子集。在 Windows 环境下提供白色与黑色两套主题，以及半透明、类毛玻璃、柔和阴影和蓝色强调色的视觉效果。

## User Stories

1. 作为 Windows 用户，我希望 Todo 常驻桌面，以便不打开完整应用也能看到任务。
2. 作为 Windows 用户，我希望插件在非编辑状态下足够安静，以便不影响当前工作。
3. 作为 Windows 用户，我希望按全局快捷键进入编辑模式，以便随时快速添加或修改任务。
4. 作为 Windows 用户，我希望点击底部隐藏图标或托盘图标进入编辑模式，以便不用键盘也能操作。
5. 作为 Windows 用户，我希望按 Escape 或点击合适区域退出编辑模式，以便快速回到干净的桌面视图。
6. 作为 Windows 用户，我希望应用记住窗口位置和尺寸，以便重启后仍保持我的布局。
7. 作为 Windows 用户，我希望可选开机自启，以便登录后立即看到任务。
8. 作为 Windows 用户，我希望插件像桌面组件而不是普通窗口，以便视觉上更自然。
9. 作为 Windows 用户，我希望界面接近 macOS Reminders，以便获得熟悉、干净的使用体验。
10. 作为 Windows 用户，我希望应用使用我提供的 SF Pro 字体，以便字体风格接近 San Francisco。
11. 作为 Windows 用户，我希望有 Today 智能列表，以便专注今天到期的任务。
12. 作为 Windows 用户，我希望有 Scheduled 智能列表，以便查看未来已安排日期的任务。
13. 作为 Windows 用户，我希望有 All 智能列表，以便查看所有未完成任务。
14. 作为 Windows 用户，我希望三个列表数量自动更新，以便快速判断今天、未来和全部任务的数量。
15. 作为 Windows 用户，我希望进入编辑模式后添加任务，以便避免查看模式下误触。
16. 作为 Windows 用户，我希望任务标题可以行内输入，以便常见操作不用打开弹窗。
17. 作为 Windows 用户，我希望任务支持备注，以便记录更细的上下文。
18. 作为 Windows 用户，我希望任务支持截止日期，以便任务自动进入 Today 或 Scheduled。
19. 作为 Windows 用户，我希望能修改或清除日期，以便计划变化时方便调整。
20. 作为 Windows 用户，我希望用圆形勾选框完成任务，以便操作直接且符合参考风格。
21. 作为 Windows 用户，我希望已完成任务默认隐藏但可查看，以便列表保持聚焦。
22. 作为 Windows 用户，我希望能恢复已完成任务，以便误操作可逆。
23. 作为 Windows 用户，我希望删除任务有确认或撤销，以便降低误删风险。
24. 作为 Windows 用户，我希望能拖拽排序任务，以便手动表达优先级。
25. 作为 Windows 用户，我希望能搜索任务，以便快速找到已有内容。
26. 作为 Windows 用户，我希望空状态显示类似 “No Reminders”，以便空列表也清晰。
27. 作为 Windows 用户，我希望查看模式和编辑模式有轻微视觉区别，以便知道当前是否可编辑。
28. 作为 Windows 用户，我希望支持键盘导航，以便高频录入更快。
29. 作为 Windows 用户，我希望可以在白色与黑色主题间切换，以便适配浅色和深色桌面环境。
30. 作为 Windows 用户，我希望切换列表时布局稳定，以便使用时不会产生明显跳动。
31. 作为 Windows 用户，我希望任务本地保存，以便离线也能使用。
32. 作为 Windows 用户，我希望未来支持导出或备份，以便数据不被锁死在应用里。
33. 作为 Windows 用户，我希望能配置快捷键、开机启动、透明度和窗口行为，以便适配自己的桌面习惯。
34. 作为 Windows 用户，我希望字体文件缺失时应用仍可用，以便迁移到其他机器也不会崩溃。
35. 作为 Windows 用户，我希望应用可以免安装运行，以便把它放在 U 盘或任意文件夹里直接使用。
36. 作为 Windows 用户，我希望不需要管理员权限也能运行核心功能，以便在公司电脑或受限 Windows 账户下使用。
37. 作为 Windows 用户，我希望可以选择数据跟随程序目录保存，以便把程序文件夹整体移动到另一台电脑后继续使用。
38. 作为 Windows 用户，我希望普通运行时数据和程序分离、便携运行时数据跟随程序目录，以便兼顾安全升级和随身携带。
39. 作为 Windows 用户，我希望应用能清晰处理字体、配置和数据路径，以便迁移后知道哪些文件需要随身携带。
40. 作为 Windows 用户，我希望开机启动等系统集成功能失败时不影响任务查看和编辑，以便便携场景下仍可稳定使用。

## Functional Requirements

### 桌面窗口行为

- 应用必须是 Windows 桌面应用，不依赖浏览器标签页运行。
- 应用必须支持免安装或绿色版形态，用户应能从任意可写文件夹直接运行。
- 应用核心功能不应要求管理员权限；缺少管理员权限时，任务查看、编辑、持久化、快捷键和托盘入口仍应尽量可用。
- 应用必须提供无边框、圆角的插件窗口。
- 应用必须支持查看模式和编辑模式。
- 应用必须支持可配置的全局快捷键进入编辑模式。
- 应用必须支持通过底部隐藏图标、系统托盘图标或二者进入编辑模式。
- 应用必须记住窗口位置、尺寸、上次选中的列表和基础偏好。
- 应用应该支持开机自启，但该能力必须是可选的，并且在无权限、便携盘路径不可用或系统策略限制时优雅降级。
- 应用必须优先实现桌面最底层行为：位于桌面壁纸和图标之上、普通应用窗口之下，不抢占正常窗口层级。
- 首版不提供置顶模式，窗口层级应服务于桌面最底层插件定位。

### 便携运行

- 应用必须支持从非安装目录、U 盘、移动硬盘或普通用户文件夹启动。
- 应用必须区分程序文件、用户数据、配置文件、字体资源和日志文件。
- 应用必须明确区分普通模式和便携模式：普通模式默认使用当前 Windows 用户的应用数据目录；便携模式默认使用程序目录下的用户数据子目录。
- 应用应该通过便携包、配置开关或便携标记文件启用便携数据模式；启用后，任务数据、配置、窗口状态和日志写入程序目录下的用户数据子目录。
- 便携数据模式必须优先检查程序目录是否可写；不可写时应回退到系统应用数据目录，并向用户显示明确状态。
- 应用应该提供数据位置查看能力，让用户知道当前任务数据实际保存在哪里。
- 应用应该支持从系统应用数据目录和便携数据目录之间迁移数据，但迁移必须由用户明确触发。
- 应用不应把用户任务数据写入安装包、可执行文件所在根目录或 Windows 受保护目录。
- 应用在不同 Windows 机器之间迁移时，应该以相对路径或可重新解析路径保存便携资源，避免硬编码原机器的绝对路径。
- 应用在 U 盘盘符变化时，便携模式下的数据和字体仍应能根据程序目录重新定位。

### 模式切换

- 查看模式必须优先保证低打扰和快速浏览。
- 编辑模式必须暴露添加、修改、完成、删除和搜索能力。
- 应用必须通过视觉状态让用户知道当前是否处于编辑模式。
- Escape 应该退出编辑模式。
- 应用应该支持通过关闭 / 最小化控件或点击非编辑区域回到查看模式。

### 任务管理

- 任务必须包含标题。
- 任务应该支持备注。
- 任务应该支持截止日期。
- 任务必须支持完成状态。
- 任务根据日期和完成状态出现在 Today、Scheduled、All 三个固定列表中。
- 应用必须支持新增、编辑、完成、恢复、删除、排序和设置日期。
- 应用应该对近期破坏性操作支持撤销。

### 列表与智能列表

- 应用必须内置 Today、Scheduled、All 三个固定列表。
- 智能列表必须动态显示未完成任务数量。
- 首版不提供自定义列表创建、重命名或删除能力。
- 侧边栏应该包含搜索框和 Today、Scheduled、All 三个固定列表入口。

### 搜索

- 搜索必须匹配任务标题。
- 搜索应该匹配备注。
- 搜索应该在输入时即时更新结果。
- 搜索应该尽量保留用户当前列表上下文。

### 视觉设计

- 应用应该采用 macOS Reminders 风格方向：白色与黑色背景、蓝色选中态、克制的灰色侧栏、大标题、紧凑任务行、圆形完成按钮、圆角卡片和柔和阴影。
- 应用必须提供可持久化的白色、黑色主题切换，并保证两套主题中的正文、次要文字、输入框、交互状态和边界线均可辨识。
- 英文、数字和拉丁标点必须优先加载从用户提供字体中裁剪的四个 WOFF2 资源：`SF-Pro-Display-Bold-latin.woff2`、`SF-Pro-Display-Regular-latin.woff2`、`SF-Pro-Text-Bold-latin.woff2`、`SF-Pro-Text-Regular-latin.woff2`。
- 中文必须优先使用系统 `PingFang SC`；系统缺失时使用内置 `Noto-Sans-SC-zh-en.woff2` 简体中文字体子集。内置子集只保留中文、英文、数字、中文标点和全角字符，不包含日文假名或韩文谚文。
- SF Pro 不含中文字形，不得把中文错误标记为由 SF Pro 渲染；中英文通过 CSS 字体 fallback 自动选择对应字形。
- 应用必须提供字体缺失时的 fallback 字体栈。
- 字体路径应该支持程序目录内置字体、用户指定字体目录和系统字体 fallback；便携模式下优先从程序目录的字体子目录解析。
- 应用应该避免直接复制 macOS 专有图标，可使用开源图标或自绘简洁符号。
- UI 首先适配参考图接近的大尺寸桌面插件，同时在更小或更大的尺寸下保持可用。

### 数据持久化

- 应用必须本地保存数据。
- 数据必须包含任务、设置、窗口状态和 schema version。
- 应用应该支持后续 schema 迁移。
- 普通模式下，应用默认应该把用户数据存储在系统应用数据目录，而不是程序目录。
- 便携模式下，应用必须支持程序目录内的便携数据目录作为默认存储位置。
- 数据文件应该采用可迁移的本地格式，并避免依赖当前机器用户名、盘符或绝对路径。
- 应用应该对数据写入失败、目录不可写、移动盘拔出和文件被占用给出可恢复处理。

### 设置

- 用户应该可以配置全局快捷键。
- 用户应该可以在白色、黑色主题间切换，选择必须在重启后保留。
- 用户应该可以开启或关闭开机启动。
- 开机启动设置必须显示当前状态：已启用、不可用、启用失败或路径失效。
- 用户不需要配置置顶模式；窗口层级应默认锁定为桌面最底层。
- 用户应该可以调整透明度或背景强度，以适配不同桌面壁纸。
- 用户应该可以选择是否显示已完成任务。
- 用户应该可以查看当前运行形态：普通模式、免安装运行或便携数据模式。
- 用户应该可以查看并切换数据存储位置，但切换前必须提示是否迁移现有数据。

## Implementation Decisions

- 产品应实现为原生感较强的 Windows 桌面应用。候选技术包括 Electron、Tauri、Flutter、.NET/WPF；最终选择应依据透明窗口质量、托盘支持、全局快捷键、打包复杂度和开发效率决定。
- 技术选型必须把便携运行作为一等约束，优先评估单目录发布、免安装运行、无管理员权限、托盘、全局快捷键、透明窗口和桌面最底层绑定的组合可行性。
- 应用应按领域拆分：窗口外壳、任务数据、固定列表过滤、设置、持久化、路径解析、快捷键处理和视觉组件。
- 桌面外壳负责桌面最底层窗口绑定、全局快捷键注册、托盘图标、开机启动、窗口状态持久化和查看 / 编辑模式切换。
- 路径解析模块负责判断运行形态、解析程序目录、系统应用数据目录、便携数据目录、字体目录和日志目录。
- 任务领域负责新增、更新、完成、恢复、删除、排序和日期设置。
- Today、Scheduled、All 三个固定列表应由任务元数据计算得出，不应存储为用户可编辑列表。
- UI 应把 macOS Reminders 作为风格参考，但保持为原创 Windows 应用。
- 字体加载应显式经过路径解析模块，并在字体缺失、路径变化或移动盘盘符变化时正常 fallback。
- 首版数据存储必须带 schema version，避免后续迁移困难。
- 首版必须明确数据位置优先级：便携包、便携标记或用户显式选择的便携数据目录优先；否则使用系统应用数据目录；如果目标目录不可写，回退到可写目录并记录状态。
- 开机启动实现不应依赖管理员权限；如果所选框架只能通过受限注册表或系统任务实现，应提供失败提示并保持应用其他功能可用。
- 删除任务等破坏性动作需要撤销或确认机制。
- 第一版优先保证本地可靠性，不做账号、云同步和跨设备能力。

## Testing Decisions

- 测试应关注外部行为：用户可见的任务状态、列表数量、智能列表过滤、持久化、模式切换和窗口设置，不优先测试内部实现细节。
- 最高价值测试切入点是任务 / 固定列表领域：创建任务、完成任务、设置日期、删除任务，并验证 Today、Scheduled、All 的过滤结果。
- 持久化测试需要验证一次会话写入的数据能在下一次会话加载，包括 schema version。
- 便携测试需要验证从普通文件夹、带空格路径、中文路径和模拟移动盘路径启动时，数据、配置、字体和日志路径解析正确。
- 迁移测试需要验证系统应用数据目录与便携数据目录之间切换时，现有任务、设置、窗口状态和 schema version 不丢失。
- 降级测试需要验证程序目录不可写、移动盘拔出、开机启动无权限、字体文件缺失和全局快捷键注册失败时，应用仍能给出明确状态并保留核心 Todo 功能。
- UI 测试需要覆盖主流程：添加任务、完成任务、切换固定列表、搜索、进入和退出编辑模式。
- 桌面集成测试需要覆盖桌面最底层窗口绑定、全局快捷键、托盘入口、开机启动设置和窗口位置恢复；如果框架自动化能力不足，至少保留手工 QA 清单。
- 视觉回归检查应覆盖空 Today、非空 Today、侧边栏选中态、编辑模式和搜索结果。
- 手工 QA 应在 Windows 不同显示缩放下检查，例如 100%、125%、150%。
- 手工 QA 应包含无管理员权限账户、Windows 受控目录、普通用户目录和可移动盘目录四类运行环境。

## Out of Scope

- 第一版不做云同步。
- 第一版不做移动端。
- 第一版不做多人协作、共享、评论和任务分配。
- 第一版不做自然语言日期解析，例如“明天 9 点”。
- 第一版不做复杂重复任务规则。
- 第一版不做文件附件。
- 第一版不做自定义列表。
- 第一版不做 Flagged / 标记重要。
- 第一版不追求 macOS Reminders 完整功能对齐。
- 第一版不直接复制 Apple 专有 UI 资源。
- 第一版不做跨设备实时同步；便携迁移只依赖用户移动文件夹或显式迁移数据。
- 第一版不保证在没有写权限的只读介质上完整可用，只要求给出明确错误和可用降级。

## Further Notes

- 用户提供的截图作为视觉参考，不作为逐像素复刻要求。
- 产品定位应是“桌面插件优先，完整 Todo 管理其次”。
- 最大技术风险是 Windows 桌面窗口行为：透明、圆角、类毛玻璃、桌面最底层绑定、全局快捷键和托盘集成需要尽早用技术原型验证。
- 第二个主要风险是便携运行路径复杂度：程序目录、系统应用数据目录、便携数据目录、字体目录和启动项路径必须早期定型，避免后续迁移成本过高。
- 第三个主要风险是范围膨胀。首版应聚焦本地任务捕捉和查看，不扩展到账户、日历、云同步或复杂提醒。

- 建议第二里程碑：实现本地持久化、便携数据模式、任务 CRUD、Today / Scheduled / All 固定列表和搜索。
- 建议第三里程碑：实现设置、开机启动降级、数据位置切换、撤销 / 确认、视觉细节和打包。
